# AI脚本平台 UI/UX规格文档 V1.0

### **1. 引言与整体UX目标**

本文档定义了AI脚本平台的用户体验(UX)目标、信息架构、核心用户流程、界面布局和设计决策，旨在为后续的架构设计和前端开发提供清晰、统一的指导。

#### **目标用户画像 (Target User Personas)**
* **主要画像 (C端)**: **内容创作者**。追求高效创作、渴望获得成功方法论、对工具的专业性和易用性有较高要求。
* **次要画像 (B端)**: **销售、管理员、超级管理员**。追求操作效率、信息清晰度、流程的可控性和数据的安全性。

#### **可用性目标 (Usability Goals)**
1.  **高效**: 关键交互需有明确的性能指标。
2.  **易学**: 新用户应能在3分钟内完成首次脚本生成的核心流程。
3.  **清晰**: 数据展示必须直观，关键操作需有清晰的反馈和二次确认。

#### **设计原则 (Design Principles)**
1.  **价值驱动**: 每个界面元素的设计都应服务于“帮助用户创作成功内容”或“提升运营管理效率”的核心价值。
2.  **引导而非命令**: 对于C端用户，产品应像一个智能伙伴，提供建议和指引，而不是一个复杂的工具箱。
3.  **一致性与可靠性**: 必须建立统一的设计系统（即使初期采用第三方库），确保跨端体验一致。
4.  **数据赋能，而非数据绑架**: AI提供建议，但用户拥有最终的创作控制权。

### **2. 信息架构 (Information Architecture)**

- 产品的整体架构分为公共访问区、C端用户后台和B端管理后台。B端后台为一个统一的应用程序框架，用户的角色决定其可见的菜单项。

#### **导航结构 (Navigation Structure)**

  * **C端用户后台**: 采用顶部或左侧侧边栏导航，包含链接：【仪表盘】、【我的脚本】、【IP诊断】、【账户设置】。
  * **B端管理后台**: 采用**左侧多级折叠侧边栏**。用户登录后，根据其角色动态显示其拥有的功能模块。
  * **面包屑导航 (Breadcrumbs)**: 在所有后台页面的顶部提供面包屑导航，以清晰地告知用户当前所在的位置。

### **3. 用户流程 (User Flows)**

#### **流程1: C端用户 - 生成AI仿写脚本**

  * **用户目标**: 已登录用户希望通过输入一个视频链接，快速获得一个高质量的仿写脚本。
  * **流程图**:

<!-- end list -->

```mermaid
graph TD
    A[开始] --> B[用户点击“生成新脚本”]
    B --> C[进入脚本生成页]
    C --> D[输入视频链接]
    D --> E[点击“立即生成”]
    E --> F{链接有效性验证}
    F -- 无效 --> G[提示“链接格式错误，请重新输入”]
    F -- 有效 --> H[页面显示“AI正在创作中，请稍候...”]
    H --> I{AI服务处理}
    I -- 成功 --> J[页面展示生成的脚本结果]
    I -- 失败 --> K[提示“服务暂时不可用，请稍后再试”]
    J --> L[用户可复制或保存脚本]
    L --> M[脚本自动保存至“我的脚本”列表]
    M --> N[结束]
    G --> D
    K --> E
```

  * **补充设计**: 新用户首次进入“我的脚本”页面时，应采用引导性的“空状态”设计，包含1-2个教学脚本范例和醒目的“生成第一个脚本”的操作指引。

#### **流程2: 销售人员 - 创建或认领线索并激活试用**

  * **用户目标**: 销售人员希望为自主开拓的新线索快速开启试用，或者查找并跟进由管理员分配的线索。
  * **流程图**:

<!-- end list -->

```mermaid
graph TD
    A[销售人员进入客户管理页] --> B[输入客户手机号]
    B --> C{系统数据库查重}
    C -- 手机号已存在 --> D{线索归属判断}
    D -- 已分配给他人 --> E[提示“线索已分配给销售[XXX]”]
    D -- 已分配给当前销售 --> F[进入该客户的详情页, 继续跟进]

    C -- 手机号不存在 --> G[销售人员补充客户姓名等信息]
    G --> H[点击“创建并开启试用”]
    H --> I{校验销售名下是否有可用试用码}
    I -- 否 --> J[提示“无可用试用码，请联系管理员”]
    I -- 是 --> K[系统执行操作]
    K --> K1[1. 创建新用户]
    K1 --> K2[2. 绑定到当前销售]
    K2 --> K3[3. 消耗一张试用码]
    K3 --> L[4. 用户状态直接设为“试用中”]
    L --> M[提示“客户[XXX]的3天试用已成功开启”]
    M --> N[流程结束]
    
    E --> A
    F --> N
    J --> A
```

  * **业务规则**: 系统内不存在“未分配”的线索，所有线索在创建时必须归属到具体销售。

### **5. 组件库 / 设计系统 (Component Library / Design System)**

  * **策略决策**: 第一期（MVP）为最大限度地专注于核心功能的快速实现和验证，将**采用一个成熟的第三方开源组件库（例如 Ant Design）**，并且**不做任何定制化的品牌视觉**。

-----

### **6. 品牌与风格指南 (Branding & Style Guide)**

- 以下为产品的长期品牌基调规划，**不作为MVP阶段的实现要求**。

  * **主色调 (Primary Color)**: **科技蓝**
  * **字体 (Typography)**: **阿里巴巴普惠体**
  * **图标 (Iconography)**: **采用所选第三方组件库的内置图标**

-----

### **7. 无障碍设计要求 (Accessibility Requirements)**

  * **范围决策**: 根据项目发起人的指令，第一期（MVP）产品将**不针对WCAG等无障碍设计标准进行专门的优化**。
