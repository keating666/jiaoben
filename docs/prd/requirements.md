# 2. 需求定义 (Requirements)

## 功能性需求 (Functional Requirements)
* **FR1**: 系统需为C端用户提供AI仿写与脚本生成功能。
* **FR2**: 系统需为C端用户提供账号IP诊断报告的生成与查看功能。
* **FR3**: C端用户必须能够对自己的脚本和IP诊断报告进行创建、查看和删除等管理操作。
* **FR4**: 系统必须支持四种角色（用户、销售、管理员、超级管理员）的权限管理体系。
* **FR5**: 系统需实现一个包含5个状态的用户生命周期，并通过试用码和正式激活码进行状态流转。
* **FR6**: 系统必须支持引导式注册，即管理员可配置官网首页的客服二维码，所有用户通过该渠道注册。
* **FR7**: 销售人员需能够在后台为指定客户生成IP诊断报告，并代客户使用视频链接进行仿写。
* **FR8**: 管理员需具备导入/分配销售线索、生成/查看/分配两种兑换码、以及查看全局数据看板的功能。
* **FR9**: 超级管理员需能够配置系统以对接不同的AI服务商API。
* **FR10**: 管理员需具备为C端用户生成仿写脚本的功能。
* **FR11**: 管理员需能够配置IP诊断报告中的"固定内容"模块，这些内容将应用于所有报告。
* **FR12**: 当系统为用户生成IP诊断报告时，需将管理员配置的"固定内容"与AI根据用户信息生成的"个性化内容"组合成一份完整的报告。
* **FR13**: 销售人员需能够在后台查看其名下激活码的使用状态。

## 非功能性需求 (Non-Functional Requirements)
* **NFR1 (性能)**: 系统核心API（如登录、查询用户信息）的平均响应时间应在200ms以下。
* **NFR2 (性能)**: "脚本生成服务"的处理过程必须是异步的，不能阻塞用户界面或核心API的响应，并应在合理时间内（例如：1-2分钟）返回结果。
* **NFR3 (可扩展性)**: 系统架构必须能够支持水平扩展，以应对未来用户量的增长。
* **NFR4 (安全性)**: 所有用户敏感数据（如手机号）在数据库中必须加密存储。
* **NFR5 (可用性)**: 系统的线上服务可用性目标为99.9%。
* **NFR6 (性能)**: 看板的数据统计和聚合必须通过异步处理或缓存机制完成，不允许对生产数据库进行实时、复杂的全表查询，以确保不会影响核心服务的性能。
